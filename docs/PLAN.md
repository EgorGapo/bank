# План работ (пишешь сам, этапы закрываются по очереди)

Каждый этап заканчивается работающим состоянием: `docker compose up` поднимается,
критерий этапа проверяем руками или скриптом. Не начинай следующий, пока не закрыт текущий.

## Этап 0 — Каркас (0.5–1 день)
- go.mod, структура: `cmd/api`, `internal/{transport,service,repository}`, `pkg/`, `migrations/`, `deploy/`.
- docker-compose: postgres + api (пока отдаёт только /healthz).
- slog в JSON + middleware с request_id, логирование каждого запроса.
- Makefile: run, test, lint, migrate-up/down.
- ✅ Критерий: `curl` в /healthz через докер, в логах JSON-строка с request_id.

## Этап 1 — Счета и миграции (1 день)
- goose, миграция accounts. POST /v1/accounts, GET /v1/accounts/{id}.
- pgxpool, repository-слой, единый формат ошибок API.
- ✅ Критерий: создал счёт, получил его; кривой uuid → аккуратный 400.

## Этап 2 — Deposit/withdraw + ledger + идемпотентность (2–3 дня, сердце проекта, ч.1)
- Миграции transfers + ledger_entries (append-only).
- Транзакция: лок счёта FOR UPDATE → проверка I1 → запись transfer + 1 ledger entry →
  обновление balance. Всё или ничего.
- Idempotency-Key: uniq-индекс, повтор возвращает исходный результат; продумать
  повтор во время pending.
- Unit-тесты service, интеграционные repository (testcontainers).
- ✅ Критерий: скрипт из 50 параллельных withdraw на счёт со 100 единицами —
  списалось ровно 100, остальные получили 422; повтор запроса с тем же ключом не дублирует.

## Этап 3 — Transfer (1–2 дня, сердце проекта, ч.2)
- Перевод: лок ДВУХ счетов в детерминированном порядке (сортировка по id), двойная проводка.
- История: GET history с курсорной пагинацией по ledger.
- ✅ Критерий: нагрузочный скрипт со встречными переводами A→B/B→A в 100 горутин —
  без клиентских ошибок 5xx, `SUM(ledger)=0`, балансы сходятся с ledger.

## Этап 4 — Outbox + Kafka + notifier (2–3 дня)
- Таблица outbox; запись события в ТОЙ ЖЕ транзакции, что и операция.
- Relay: батчи → Kafka acks=all → sent_at. Kafka в compose.
- Сервис notifier: consumer group, идемпотентность по event_id, ручной commit после записи,
  DLQ после N ретраев.
- ✅ Критерий: kill -9 api между операцией и отправкой → после рестарта событие доехало;
  дубль события не создал дубль уведомления.

## Этап 5 — Observability (1–2 дня)
- Prometheus-метрики (RED + бизнес + outbox/consumer lag), OTel-трейсы через оба сервиса
  (контекст через Kafka headers), Grafana с дашбордом, всё в compose.
- ✅ Критерий: в Jaeger/Tempo виден один трейс HTTP→БД→Kafka→notifier;
  на дашборде живой RPS и лаги.

## Этап 6 — Nginx + нагрузка + graceful shutdown (1–2 дня)
- nginx перед api: rate limit, таймауты, X-Request-Id.
- Graceful shutdown обоих сервисов (проверить: под нагрузкой SIGTERM → ноль оборванных запросов).
- k6-сценарии: ровная нагрузка 1000 RPS и hot-account. Прогнать, снять p99, потюнить
  (индексы, размер пула, батчи relay). Результаты — в README.
- ✅ Критерий: цифры из SPEC §6 достигнуты, инварианты после прогона целы.

## Этап 7 — Полировка (по желанию)
- golangci-lint строгий, CI, README с архитектурной схемой и обоснованием решений
  (изоляция, порядок локов, ключи партиционирования) — это то, что спрашивают на собесах.
