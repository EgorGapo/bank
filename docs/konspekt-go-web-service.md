# Go веб-сервис: конспект по мини-банку

Конспект по вопросам, разобранным при написании ledger-сервиса. Проект: транзакции, идемпотентность, outbox+Kafka, observability. ТЗ — SPEC.md, план — PLAN.md.

--- **При люой ощибки в транзакции она умирает и все вызовы через tx уже не работают**

## 1. Архитектура и слои

### Слои и направление зависимостей
```
transport (controller, middleware, route)  →  usecases  →  storage  →  Postgres
```
- **controller** — парсинг/валидация запроса, выбор HTTP-статуса. Без бизнес-логики.
- **usecases** — бизнес-логика, решения (генерация id, статусы, транзакции).
- **storage** — только SQL. Тупой исполнитель: «дали структуру — записал».
- Всё непубличное — в `internal/` (компилятор запрещает внешние импорты).

### Интерфейсы объявляет потребитель
- НЕ в пакете реализации (`storage.Storage` в storage — неидиоматично).
- Usecases объявляет у себя `type Storage interface {...}` — что ЕМУ нужно от хранилища. Controller объявляет `type Bank interface {...}`.
- Следствие: usecases не импортирует storage вообще. Связывает всё только `app.Run` (composition root): `storage.NewPostgres → usecases.NewBank → controller.New`.
- Профит: маленькие интерфейсы, лёгкие моки, инверсия зависимостей = направление стрелочки импорта.

### Вертикальные срезы, а не слои целиком
После миграций писать не «весь storage», а одну фичу через все слои до работающего состояния (POST /accounts насквозь → curl). Слои проектируются от потребителя; заглушки всех методов = ноль обратной связи.

### Правила сигнатур
- UUID генерит **usecase** (не БД, не storage): id нужен до INSERT — для логов, событий outbox, ответа клиенту.
- Сохраняешь сущность целиком → передавай структуру: `CreateAccount(ctx, acc domain.Account)`. Узкая операция → поля: `GetAccount(ctx, id)`. Критерий: если в структуре заполнена половина полей — метод просит поля; если аргументов 4+ — просит структуру.
- Заглушки сразу с честными сигнатурами `(T, error)` — они вморожены в интерфейсы трёх слоёв.
- Доменная модель (`domain.Account`) — в нейтральном пакете `internal/domain`, чтобы все слои импортировали её, а не друг друга.

## 2. Ошибки

### Главный принцип
**Либо логируешь, либо возвращаешь — не оба.** Функция, у которой не получилось, возвращает error и всё. Логирует тот, кто ошибку окончательно обрабатывает.

### Нет logger.Fatal
- `log.Fatal` = печать + `os.Exit(1)`. **defer не выполняются** — пул не закроется, shutdown не случится.
- В slog Fatal нет намеренно: «умереть» — управление потоком, не логирование.
- Паттерн: `run() error`, а `os.Exit` только в main, в одном месте:
```go
func main() {
    if err := run(); err != nil { slog.Error("service failed", "error", err); os.Exit(1) }
}
```

### Оборачивание ошибок
- Стандарт: `fmt.Errorf("insert account: %w", err)` — `%w` сохраняет цепочку для `errors.Is/As`.
- `%v` вместо `%w` — рвёт цепочку (используется осознанно, чтобы отрезать внутренности).
- Sentinel-ошибки: storage ловит `pgx.ErrNoRows` → возвращает `domain.ErrAccountNotFound`. Верхние слои не знают про pgx.
- Свой тип с полями + `Unwrap()` — когда обработчику нужны данные (`InsufficientFundsError{Balance, Requested}`), достаётся `errors.As`.
- Оборачивать в usecase поверх storage? **Оборачивай, когда добавляешь информацию** (имя бизнес-операции, id, различение нескольких вызовов). Прозрачный проброс одного вызова можно не оборачивать.
- pgx v5 возвращает `pgx.ErrNoRows`, НЕ `sql.ErrNoRows` — проверять именно его.

### Конвейер ошибки запроса
```
storage: pgx.ErrNoRows → domain.ErrAccountNotFound
usecase: return err (или обернуть контекстом операции)
handler: respondError(w, r, err) → switch errors.Is → HTTP-статус + JSON
```
- Ошибка запроса НЕ доходит до main: сигнатура хендлера ничего не возвращает. Хендлер — последняя инстанция, где ошибка «умирает», превращаясь в HTTP-ответ.
- Ошибки валидации транспорта (`ErrInvalidUUID`) живут в пакете controller; бизнес-ошибки — в domain.

## 3. HTTP-статусы

- Успех выбирает **хендлер** (транспортный слой): 201 — создали сущность, 200 — остальное успешное, 204 — успех без тела.
- По умолчанию (если `WriteHeader` не вызван) уходит **200** при первом `Write`.
- Ошибки — централизованно в `respondError`: один switch на весь API. Добавить статус = одна ветка.
- 4xx vs 5xx = «кто виноват»: 4xx — клиент (повторять бессмысленно), 5xx — сервис (можно ретраить). insufficient_funds = 422 (сервис работает правильно!).
- default-ветка = всегда 500 + `logger.Error` — единственное место логирования 5xx.
- Наружу в message: свои тексты валидации — можно; внутренние ошибки — НЕТ (утечёт DSN/SQL), только константа "internal server error".
- 404/405 для несматченных роутов отдаёт сам chi (можно кастомизировать `r.NotFound(...)`).

## 4. Логирование

### slog
```go
logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{Level: lvl}))
```
- JSON в **stdout** (докер собирает сам; файлы — не дело приложения, 12-factor).
- Уровень из конфига: `lvl.UnmarshalText([]byte(cfg.Log.Level))`.
- `slog.SetDefault(logger)` — чтобы чужой `log.Printf` тоже шёл в JSON. Но в своём коде — передавать логгер явно (конструктором) или брать из контекста.
- `logger.With("k", v)` — возвращает НОВЫЙ логгер с пришитым полем; каждая запись через него содержит поле автоматически.

### Кто что логирует (одно событие = одна строка)
| Слой | Логирует |
|---|---|
| middleware AccessLog | одна строка на КАЖДЫЙ запрос: method, path, status, duration_ms |
| respondError (default) | 5xx с цепочкой ошибки — единственный носитель ПРИЧИНЫ |
| usecase | бизнес-события (Info: «перевод выполнен»), не ошибки |
| storage | ничего (ошибки возвращает; отладка SQL — через pgx QueryTracer) |
| хендлеры | ничего руками |

- 4xx не логируются: статус уже в access-логе, вторая строка — дубль. Всплеск 404 — вопрос метрик, не логов.
- 500 даёт две строки (Error с причиной + access-лог со статусом) — это НЕ дубль: разные данные, склеиваются по request_id.
- Тест перед `logger.X(...)`: «этот факт кто-то выше уже запишет?» Да → не пиши. Ошибка → верни.
- Ключи логов: ASCII snake_case (`duration_ms`), по ним фильтруют. Duration — числом (`.Milliseconds()`), не `time.Duration` (уйдёт наносекундами).
- Не логировать конфиг целиком (утечёт пароль).

### request_id vs idempotency_key
| | request_id | Idempotency-Key |
|---|---|---|
| Генерит | сервер (middleware), или берёт от nginx | **клиент** |
| При ретрае | НОВЫЙ | **тот же** — в этом вся суть |
| Зачем | наблюдаемость: склеить логи одного запроса | корректность: не выполнить операцию дважды |
| Живёт | один запрос | навсегда, в БД (UNIQUE-индекс) |

Ключ в БД (не в памяти): дедупликация должна переживать рестарт и работать между репликами. Сервер сам НЕ может понять, что два запроса — одна операция: только клиент знает, что это ретрай.

## 5. Middleware

### Паттерн
```go
func My(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // до
        next.ServeHTTP(w, r)
        // после
    })
}
```
- С зависимостью — фабрика: `func RequestID(base *slog.Logger) func(http.Handler) http.Handler`.
- Подключение: `r.Use(...)` строго ДО объявления роутов (иначе chi паникует). Порядок Use = порядок выполнения: RequestID → AccessLog.
- Вся сборка конвейера — в одном месте (пакет route). Можно вешать на под-дерево (`/v1` со своим Use, `/healthz` вне).

### RequestID
- Взять из заголовка `X-Request-Id` (пришёл от nginx) или сгенерить uuid.
- Вернуть клиенту в заголовок ответа.
- В контекст класть **логгер с пришитым полем** (`base.With("request_id", id)`), не голый id — потребители пишут в него и поле приезжает само. Голый id — вторым ключом, если нужен как значение (в Kafka-событие, в тело 500).
- Ключ контекста — свой неэкспортированный тип (`type ctxKey int`), не строка: защита от коллизий.
- `FromContext(ctx)` с фолбэком на `slog.Default()`.

### AccessLog и перехват статуса
- `http.ResponseWriter` — интерфейс из 3 методов, **write-only труба в сокет**: статус не прочитать обратно (WriteHeader мог уже улететь клиенту). Конкретный тип `*http.response` неэкспортирован.
- Решение — декоратор: встроить `http.ResponseWriter` в свою структуру и **переопределить** `WriteHeader` (запомнил код → делегировал). Встраивание без переопределения НЕ перехватывает!
- Хендлер может не вызвать WriteHeader (неявные 200) → инициализировать `statusCode: http.StatusOK`.
- Наивная обёртка теряет `http.Flusher`/`http.Hijacker` (нужны для стриминга); у chi есть `middleware.WrapResponseWriter`.

### Зачем AccessLog вообще
Глаза на сервис: фиксирует ВСЕ запросы, включая те, где бизнес-лога нет (404, валидация). Латентность (duration). «Шапка» истории запроса, к которой по request_id клеятся остальные строки. Позволяет хендлерам не логировать.

### Rate limiting — НЕ middleware (в этом проекте)
Живёт в nginx (`limit_req`): отсекать надо ДО того, как запрос занял горутину и коннект. Лимит в приложении — когда завязан на бизнес-сущность (user_id) или общий на реплики (Redis).

## 6. context.Context

### Откуда берётся контекст запроса
`Server.BaseContext` (по умолчанию `context.Background()`) → контекст TCP-соединения → **контекст запроса** (`r.Context()`). Отменяется когда: клиент отвалился / хендлер вернулся / сервер выключается.

### Матрёшка
`Background → соединение → запрос (отменяемый) → +логгер → +id`. `WithValue` создаёт производный контекст; отмена родителя видна всем детям.

### Три корня в проекте
| Контекст | Живёт | Отменяется |
|---|---|---|
| `r.Context()` | один запрос | клиент ушёл / ответ отдан |
| `signal.NotifyContext` в Run | весь аптайм | SIGTERM / Ctrl+C |
| `Background()` в мигаторе | один прогон | никогда |

Правило: код запроса — на контексте запроса; фоновое (outbox-relay) — на контексте приложения. Фоновая горутина на `r.Context()` умрёт с ответом — классическая ошибка.

### Контекст как ограничитель ожидания
`pool.Ping(pingCtx)` — контекст не «раздаётся» БД, а ограничивает ожидание вызова. Без дедлайна Ping ждёт TCP-таймаут ОС (минуты). Паттерн:
```go
pingCtx, cancel := context.WithTimeout(ctx, 5*time.Second) // родитель — живой ctx: сигнал тоже прервёт
```
Инвариант: в приложении нет ни одного неограниченного ожидания (Ping 5с, Server Read/WriteTimeout, shutdown 10с). Запросы storage наследуют контекст запроса — им свой таймаут не нужен.

### Отмена посреди денежной транзакции
Клиент оборвался, когда транзакция уже «решена»? Варианты: откатить (клиент повторит по идемпотентности) или дожать через `context.WithoutCancel` (Go 1.21+). Выбор — осознанно, в README.

## 7. Как работает HTTP-сервер в Go

### Две фазы
1. **Сборка (один раз)**: config → logger → pool → store → usecase → ctrl → router → `ListenAndServe()` (блокируется).
2. **Обслуживание (на каждый запрос)**: Accept → **горутина на соединение** → парсинг запроса → `router.ServeHTTP` → цепочка middleware → хендлер → ответ.

### Следствие: конкурентность
Один объект controller/usecase/storage — тысячи одновременных вызовов его методов.
- В полях структур — только созданное при старте и неизменяемое (логгер, пул, соседний слой).
- Всё per-request — в аргументах: ctx, w, r. Поэтому ctx передаётся параметром, а не лежит в структуре.
- Разделяемое должно быть потокобезопасно: pgxpool, slog.Logger — да.
- Модель: слои = цех со станками (общие), запрос = деталь (своя у каждой горутины), данные детали едут с ней.

### *http.response
Неэкспортированная структура net/http, реализует ResponseWriter. Создаётся на КАЖДЫЙ запрос (шаг 4 цикла), живёт миллисекунды. При keep-alive соединение живёт долго, но на каждый запрос — новый response. Хендлер видит её только через маску интерфейса.

### Keep-alive
Одно TCP-соединение обслуживает много запросов подряд — та же горутина в цикле читает следующий запрос.

## 8. Graceful shutdown

### Паттерн
```go
ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
defer stop()
...
errCh := make(chan error, 1)                    // буфер: иначе горутина утечёт
go func() { errCh <- srv.ListenAndServe() }()

select {
case err := <-errCh:                             // сервер не поднялся (порт занят)
    return fmt.Errorf("server: %w", err)
case <-ctx.Done():                               // пришёл сигнал
    shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()
    return srv.Shutdown(shutdownCtx)
}
```

### Что делает Shutdown под капотом
1. Закрывает листенер — новые соединения не принимаются; `ListenAndServe` возвращает `http.ErrServerClosed`.
2. Работающие хендлеры НЕ трогает — их контексты не отменяются, запросы доделываются.
3. В цикле ждёт освобождения всех соединений → nil (мягкий успех).
4. `shutdownCtx` — предел терпения пункта 3: через 10с возвращает `DeadlineExceeded`, процесс выходит, остатки умирают.

### Ключевые «почему»
- `shutdownCtx` хендлерам не раздаётся — это входной параметр ожидания самого Shutdown (как ctx у Ping).
- Родитель — `Background()`, не `ctx`: ctx в этой ветке УЖЕ отменён; производный от него таймаут был бы мёртвым и Shutdown вышел бы мгновенно, никого не дождавшись.
- Без `signal.NotifyContext` Ctrl+C убивает процесс мгновенно (дефолт рантайма), ветка ctx.Done() не наступает.
- Shutdown работает в паре с ограничением длины запросов (Read/WriteTimeout): 30-секундный хендлер он не спасёт.
- Проверка: sleep в хендлер → curl → Ctrl+C → ответ должен доехать.
- `os.Exit` где-либо кроме main ломает всё это (defer не выполняются).

## 9. PostgreSQL и миграции

### Типы и схема
- Деньги: **bigint** минорные единицы (копейки). Никаких float/numeric в этом проекте.
- Время: **timestamptz** (абсолютный момент, хранится UTC). `timestamp` без зоны — «наивные» цифры, поплывут при смене зоны. `TIMESTAMPZ` — опечатка, нет такого типа (timestamp + **tz** = time zone).
- Статусы: text + CHECK constraint, не ENUM: `status TEXT NOT NULL DEFAULT 'active' CONSTRAINT accounts_status_check CHECK (status IN (...))`. Менять список — DROP/ADD CONSTRAINT без сюрпризов; у ENUM `ALTER TYPE ADD VALUE` капризен в транзакциях, значения не удалить.
- Констрейнтам — имена (`accounts_balance_non_negative`): безымянные получат автогенерированные, менять больно.
- CHECK `balance >= 0` — БД как последний рубеж инварианта, даже если код ошибся.
- uuid генерит приложение → DEFAULT в БД не нужен; `uuid-ossp` не нужен вообще (с PG13 есть `gen_random_uuid()`).

### SQL-грабли (пойманные)
- `INSERT INTO t VALUES (...)` без списка колонок — значения лягут по порядку колонок таблицы. Всегда: `INSERT INTO t (col1, col2) VALUES ($1, $2)`.
- `RETURNING (a, b)` — скобки делают ОДНУ составную запись, Scan в две переменные ломается. Без скобок: `RETURNING a, b`.
- `Scan(&x, &y)` — только указатели.
- Один INSERT атомарен сам по себе — транзакция не нужна. Нужна, когда несколько statement'ов должны выполниться все-или-никак.

### goose
- `-- +goose Up` / `-- +goose Down`; plpgsql-функции — в `+goose StatementBegin/End` (иначе порежет по `;`).
- Down откатывает ВСЁ, что сделал Up, в обратном порядке. `DROP TABLE` снесёт триггер, но НЕ функцию — её отдельным `DROP FUNCTION`.
- Тест каждой миграции: `up → down → up` без ошибок.
- Миграции по мере этапов, не всё сразу: эволюция схемы — то, ради чего миграции существуют.
- Служебная таблица `goose_db_version` — что уже применено (даёт идемпотентность повторного прогона).

### Мигратор — отдельный бинарник (cmd/migrate), не при старте api
Причины:
1. **Реплики**: N инстансов наперегонки катают схему при каждом деплое.
2. **Управляемость**: сломанная миграция = понятная ошибка отдельного шага, а не crash-loop всего флота.
3. **Права**: мигратор — DDL-пользователь, приложение — только DML.
4. Рестарт ≠ деплой: рестарты не должны трогать схему.

Порядок деплоя в compose:
```yaml
migrate:
  depends_on: { postgres: { condition: service_healthy } }
api:
  depends_on: { migrate: { condition: service_completed_successfully } }
```
Миграция — шаг деплоя; api — долгоживущий процесс. В k8s то же самое = Job/init-container.

Каждый процесс создаёт СВОЁ подключение к БД (общей памяти нет). Мигратору пул избыточен (одно соединение, последовательно) — достаточно `sql.DB`.

### Нейминг
- Таблицы: множественное число (`accounts`, `transfers`) — главное, единообразно. `outbox` — имя паттерна, можно ед.ч.
- Всё snake_case строчными. Констрейнты: `<table>_<column>_<suffix>`.
- Файлы миграций: единый шаблон `NNN_create_<table>.sql`.

## 10. Схема БД проекта (5 таблиц)

- **accounts**: id (uuid PK), status, balance (bigint, CHECK >= 0), created_at, updated_at. Баланс — денормализация для скорости; истина — в ledger.
- **transfers** (операции deposit/withdraw/transfer): id, **idempotency_key UNIQUE**, type, from_account_id (NULL у deposit), to_account_id (NULL у withdraw), amount (CHECK > 0, без знака — направление задаёт type), status (pending/completed/failed), error_code, created_at, completed_at. CHECK на согласованность type ↔ NULL-поля; `from <> to`. Жизнь строки: INSERT pending (ловится дубль ключа) → проводки → UPDATE completed/failed.
- **ledger_entries** (append-only, никогда UPDATE/DELETE): id (bigserial = курсор пагинации), transfer_id, account_id, amount (СО ЗНАКОМ), balance_after, created_at. Индекс (account_id, id) под историю. Инвариант: SUM(amount) = 0.
- **outbox**: id (= event_id для дедупликации consumer'ом), topic, key, payload jsonb, created_at, sent_at NULL. Частичный индекс WHERE sent_at IS NULL.
- **notifications** (notifier): event_id UNIQUE — идемпотентность consumer'а.

## 11. Хендлеры и DTO

### Анатомия хендлера
```
распарсить/провалидировать → вызвать usecase с r.Context() → ошибка? respondError + RETURN → успех? DTO + respondJSON(201)
```
Забытый `return` после respondError = второй ответ поверх первого.

### DTO ≠ доменная модель
На каждую ручку с телом — свой request/response тип с json-тегами (живут в controller):
- доменная структура отдаст в JSON всё, включая лишнее, и с именами полей Go (`"ID"`);
- API-контракт застывает, домен меняется — связав их, каждый рефакторинг = breaking change;
- ручки без тела (GET по id) — без request-DTO.
Хендлер = переводчик между «как сервис говорит с миром» и «как думает внутри». Стиль ключей JSON (snake_case) — единый по всем DTO.

### Хелперы
- `respondJSON(w, status, v)` — Content-Type, WriteHeader, Encode.
- Ошибка Encode игнорируется осознанно (`_ =`): статус уже улетел, исправить нельзя; падает либо из-за несериализуемого типа (ловится первым тестом), либо клиент отвалился (некому отвечать). Строже: Marshal в буфер ДО WriteHeader.
- `http.Error` — для plain text; у нас JSON-формат → свой хелпер.
- `decodeJSON` с `http.MaxBytesReader` (не принять гигабайт) — для ручек с телом.

## 13. Авторизация и микросервисы (теория, вне MVP)

### Аутентификация ≠ авторизация
- Аутентификация — «кто ты» (пароли, токены). Живёт в auth-сервисе/gateway, НЕ в ledger.
- Авторизация владения — «твой ли счёт». Живёт в ledger: колонка owner_id + проверка. owner_id — внешний id, таблица users НЕ нужна (чужая зона ответственности).

### Путь запроса с auth
```
логин (редко): клиент → auth: пароль → bcrypt-проверка → подписанный JWT {user_id, exp}
запрос: клиент → nginx: rate limit → auth_request (валиден токен?) → X-User-Id: 42 → api
api: request_id → валидация → владение (403?) → идемпотентность → транзакция
```
- JWT stateless: подпись проверяется локально публичным ключом, auth не вызывается; цена — не отозвать мгновенно (потому живёт 5–15 мин).
- Opaque + introspection: отзыв мгновенный, но auth — узкое место.
- api слепо верит X-User-Id, потому что недоступен снаружи (нет проброшенного порта, только внутренняя сеть).
- Плохой токен отсекается на nginx — api и БД не потревожены.

### Ledger как ядро банка
Текущий сервис = core ledger. Вокруг добавляются, не трогая его код: auth, customer/profile (KYC), payments-orchestrator (саги, hold/capture/release), antifraud (подписан на Kafka-топик), limits/compliance, statements (CQRS read-модели), cards. Тест хорошей архитектуры: система растёт добавлением сервисов, а не переписыванием ядра.
